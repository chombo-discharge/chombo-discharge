ROOT CAUSE 1 — BRANCH (if inside the lambda)
---------------------------------------------
GCC reason: "not vectorized: control flow in loop."

The most widespread reason. Any lambda that contains an if/else — most
commonly branching on ebisbox.isRegular(iv), ebisbox.isCovered(iv), or a
tolerance check — blocks vectorization.

The if (ebisbox.isRegular(iv)) pattern cannot be removed by a loop change —
it exists because the same Box loop handles both regular and cut cells.
The structural fix would be to restrict Box loops to all-regular patches
(guarded by ebisbox.isAllRegular()) and handle cut cells separately through
a VoFIterator path. That is already the design intent in most solvers, but
many lambdas still mix both cases in a single Box loop.


ROOT CAUSE 2 — INNER for LOOP INSIDE THE LAMBDA
-------------------------------------------------
GCC reason: "loop nest containing two or more consecutive inner loops" /
            "multiple nested loops."

Lambdas that contain a for (int d = 0; d < SpaceDim; ++d) loop (direction
loops for flux summation, norm computation, coarsening averages) produce a
nested loop inside the BoxLoops i-loop. GCC refuses to vectorize the outer
i-loop when a non-trivial inner loop is present.

The three-level loop nest (outer j, inner i from BoxLoops template, inner d
in lambda) is the fundamental barrier. Template-unrolling the d-loop (using
D_TERM/D_DECL to expand explicitly for SpaceDim=2 or 3) would resolve this
but requires restructuring the lambda.


ROOT CAUSE 3 — RAII OBJECT WITH DESTRUCTOR IN SCOPE
----------------------------------------------------
GCC reason: "statement can throw an exception: VoFIterator::operator" /
            "FaceIterator::reset"

When a VoFIterator or FaceIterator is constructed in the enclosing scope of
the lambda — even if the lambda itself never uses it — GCC must conservatively
assume that instructions inside the vectorized loop could trigger an unwind
that calls the iterator's destructor. This creates implicit control flow and
blocks vectorization.

  File                                call sites affected
  CD_EBHelmholtzOp.cpp                ~614/618: VoFIterator vofit(...) is live
                                       in the enclosing if-block when the Box
                                       loop runs
  CD_EddingtonSP1.cpp                  FaceIterator in enclosing scope

Fix: wrap the BoxLoops call in its own { } block so that the iterator's
lifetime ends before the loop starts, or reorder the code so the Box loop
executes before the iterator is constructed.


ROOT CAUSE 4 — TOO COMPLEX TO ATTEMPT (SILENT)
-----------------------------------------------
GCC emits NO missed reports — it does not even try to vectorize.

Occurs when lambdas call functions not visible to the inliner: virtual
dispatch through Chombo base class pointers, calls to opaque library
functions, or initialization lambdas passed as std::function. GCC abandons
analysis without emitting a diagnostic.

CONCLUSIONS
-----------
1. The 7% that vectorize are already the right ones: pure arithmetic inner-loop
   updates that dominate actual solver time (5-point stencil, flux divergence,
   scalar multiply).

2. Root causes 1-3 are structural to the EB algorithm. The isRegular/isCovered
   branching is fundamental; SpaceDim direction loops are fundamental. These
   cannot be eliminated without algorithmic changes.

3. Root cause 3 (exception scope) is the only immediately fixable case. Moving
   VoFIterator construction outside the scope of the Box loop removes the GCC
   exception-safety barrier and would allow those 2-3 call sites to vectorize.

4. Root cause 4 (silent) is limited by Chombo library opacity. No action is
   possible without making the called functions header-inlineable.

5. The compile-time stride template (BoxLoops::loop<D_DECL(Si,Sj,Sk)>) works
   as intended for the color Gauss-Seidel kernel (stride 2, confirmed AVX2) and
   for all unit-stride kernels that are structurally vectorizable.
